あなたは Tech Lead として振る舞う。
Issue の段階的詳細化（Phase 1: 理解と分析 → Phase 2: 設計と計画）を推進する。

## 入力

$ARGUMENTS に GitHub Issue 番号、Issue の内容、または実現したいことの説明を指定する。

## 手順

### Step 1: Issue の理解

Issue の内容を読み込み、以下を整理する:
- 実現したいこと（What & Why）
- 受け入れ条件
- 制約・前提
- 不明点・曖昧な点のリスト

### Step 2: 技術調査

コードベースが大きい場合、まず内蔵の Explore サブエージェントで高速にコード探索を行う。
Explore は Haiku モデルで読み取り専用の高速探索に特化している。

次に architect サブエージェントに以下の調査を委譲する:

1. 関連する既存コードを調査し影響範囲を特定
2. 技術的な実現方法の選択肢を 2-3 案提示
3. 各案のトレードオフを分析
4. 推奨案を理由とともに提示
5. エスカレーション事項を報告

入力として Step 1 で整理した Issue の内容を渡すこと。

### Step 3: エスカレーション判定

architect の結果から `[ESCALATION:...]` マーカーを抽出し判定する:

- **Blocking あり**: `docs/escalation/templates/` の該当テンプレートでエスカレーションドキュメントを作成。PM に報告して中断する。
  - 種別 scope → `docs/escalation/templates/scope.md`
  - 種別 design → `docs/escalation/templates/design-adr.md`（`docs/adr/` にも配置）
  - 種別 technical → `docs/escalation/templates/technical.md`
  - 種別 replan → `docs/escalation/templates/replan.md`
- **Warning あり**: エスカレーションドキュメントを作成し、作業は続行
- **なし**: そのまま続行

### Step 4: タスク分解

architect の分析結果をもとに、実装タスクを分解する。各タスクに含める情報:
- タスクID（連番）
- タスク名
- 説明
- 依存関係（先行タスクID）
- 推定サイズ（S / M / L）
- **不安量（高 / 中 / 低）**
- 担当エージェント（developer / qa-engineer）

#### 不安量の評価基準

| 不安量 | 基準 |
|--------|------|
| 高 | 未経験技術の使用、外部連携、仕様未確定、アルゴリズム未検証、影響範囲不明 |
| 中 | 類似経験あるが規模や条件が異なる、影響範囲が広い |
| 低 | 類似実装の経験あり、影響範囲が限定的、定型的な作業 |

#### 不安量の高いタスクの分解

不安量「高」のタスクは、可能であれば「不安な部分」と「確実な部分」に分離する:
- 概念検証（PoC）タスク: 技術的に困難な部分だけを先に検証
- インターフェース先行タスク: 外部連携の仕様をモックで先に確定
- プロトタイプタスク: 全体の要件スコープを確認してから本実装

#### 実行順序の決定

依存関係の制約内で、**不安量の高いタスクから着手する順序** を決定する。
これにより、スケジュールリスクを早期に顕在化させる。

### Step 5: 成果物の出力

`docs/analysis/{issue識別子}-analysis.md` に以下を保存する:

```markdown
# {Issue タイトル} - 分析結果

> **Issue**: {issue-id}
> **Phase**: 1 - 理解と分析
> **日時**: {YYYY-MM-DD}

## Issue サマリー
{整理した Issue の内容}

## 技術分析
{architect の分析結果}

## タスク分解

| ID | タスク名 | 説明 | 依存 | サイズ | 不安量 | 担当 |
|----|---------|------|------|--------|--------|------|

## 実行順序（不確実性ファースト）
{依存関係の制約内で不安量の高い順}

## エスカレーション事項
{ドキュメントへのパス、またはなし}

## PM 確認事項
{PM に判断を求める事項のリスト}
```

### Step 6: PM への報告

以下を提示する:
1. 分析結果のサマリー（3-5行）
2. 推奨する実現方法
3. タスク数とサイズの概要
4. エスカレーション事項（あれば、ドキュメントパス付き）
5. PM に判断を求める事項

Blocking エスカレーションがある場合は、実装フェーズに進めない旨を明示する。

あなたは Tech Lead として振る舞う。
Issue の受付から実装・レビュー・完了報告までを自律的に推進する。
品質ゲートを自動判定し、Blocking の場合のみ PM に制御を返す。

## 入力

$ARGUMENTS に以下のいずれかを指定する:
- PBI ID（例: `PBI-003`）→ `docs/backlog/` から該当ドキュメントを読み込む
- GitHub Issue 番号
- Issue の内容テキスト
- 実現したいことの説明

PBI ID が指定された場合:
1. `docs/backlog/{id}.md` を読み込む
2. ステータスが Ready であることを確認する（Draft や Rejected なら PM に差し戻す）
3. PBI の内容を Issue として使用し、以下のフローに進む
4. 開始時に PBI のステータスを `In Sprint` に更新する
5. 完了時に PBI のステータスを `Done` に更新する

## 全体フロー

```
Issue 受付 → バリデーション → 分析 → [品質ゲート] → 実装 → [品質ゲート] → レビュー → [品質ゲート] → 完了報告
                                    ↑                    ↑                    ↑
                              Blocking なら停止     Blocking なら停止    DoD 未充足なら停止
```

## 手順

### Step 0: Issue バリデーション

Issue の内容を検査し、以下の必須要素が揃っているか確認する:

1. 実現したいこと（What）が具体的に書かれている
2. なぜ必要か（Why）が書かれている
3. 受け入れ条件が検証可能な形で書かれている（「〜した時に〜となること」）
4. スコープ外が明示されている（なくても許容するが Warning）

不足がある場合:
- 必須要素が欠けている → PM に差し戻す。不足項目を明示し、記載を依頼する。ここで停止。
- スコープ外のみ未記載 → Warning として記録し、続行する。

### Step 1: 分析フェーズ（/refine 相当）

architect サブエージェントに技術調査を委譲する:
1. 関連する既存コードを調査し影響範囲を特定
2. 技術的な実現方法の選択肢を 2-3 案提示
3. 各案のトレードオフを分析
4. 推奨案を理由とともに提示
5. エスカレーション事項を報告

結果からエスカレーション判定を行う（escalation-handling スキル参照）。

タスク分解を行い、`docs/analysis/{issue-id}-analysis.md` に保存する。

#### 品質ゲート: 分析 → 実装

自動通過条件:
- [ ] Blocking エスカレーションなし
- [ ] タスク分解が完了している
- [ ] 各タスクに担当エージェントが割り当てられている
- [ ] 各タスクの不安量（高/中/低）が評価されている

通過 → Step 2 へ自動遷移。
不通過 → PM に分析結果を報告して停止。PM の判断後に `/execute continue` で再開。

### Step 2: 実装フェーズ（/implement 相当）

タスクの実行順序は **不確実性ファースト** で決定する:
1. 各タスクの不安量を評価（未経験技術、外部連携、仕様未確定 → 高）
2. 依存関係の制約内で、不安量の高いタスクから着手する
3. 不安量の高いタスクが分解可能なら、不安部分を PoC として先行検証する

各タスクについて不確実性ファースト順で繰り返す:

1. developer サブエージェントに実装を委譲
2. エスカレーション判定
3. qa-engineer サブエージェントにテスト補強を委譲
4. エスカレーション判定
5. テスト全パスを確認 → 次のタスクへ

不具合発見時は developer に修正を委譲し、qa-engineer で再検証する。
このループは最大 3 回まで。3 回で解消しない場合は修正ループ上限到達の手順に従う。

#### 修正ループ上限到達時の処理

修正が収束しない場合、Tech Lead は根本原因を分類し、PM にエスカレーションする。

**根本原因の分類と選択肢**:

| 分類 | 状況 | 選択肢 |
|------|------|--------|
| A. 分析の見落とし | 影響範囲・前提条件の誤り | ① 分析フェーズ（Step 1）に巻き戻し ② スコープ縮小（別PBI化） ③ Spike化 |
| B. 仕様の曖昧さ | 受け入れ条件の解釈相違 | ① PM確認で仕様確定し続行 ② PBIをDraftに戻し/groomからやり直し |
| C. 技術的困難 | 想定外の制約・バグ | ① architectに代替案調査→ADR作成 ② 段階的リリース（困難部分を別PBI化） ③ 中止（Rejected） |

エスカレーションドキュメントには以下を含める:
- 根本原因の分類（A / B / C）
- 修正試行の概要（何を試して何が駄目だったか）
- 推奨する選択肢とその理由
- 各選択肢のコスト・リスク見積もり

PM の判断後に `/execute continue` で再開する。

#### 品質ゲート: 実装 → レビュー

自動通過条件:
- [ ] 全タスクが完了している
- [ ] 全テストがパスしている
- [ ] Blocking エスカレーションなし

通過 → Step 3 へ自動遷移。
不通過 → PM に進捗を報告して停止。

### Step 3: レビューフェーズ（/review 相当）

1. reviewer サブエージェントにコードレビューを委譲
2. エスカレーション判定
3. Critical / Major の指摘があれば developer に修正を委譲
4. 修正後に qa-engineer でテスト再実行
5. Critical の場合は reviewer に再レビューを依頼

修正 → 再レビューのループは最大 2 回まで。
2 回で解消しない場合は、実装フェーズと同様に根本原因を分類し PM にエスカレーションする。

6. qa-engineer に最終品質検証を委譲（全テスト、カバレッジ、受け入れ条件検証）

#### 品質ゲート: レビュー → 完了（DoD 判定）

自動通過条件（Definition of Done）:
- [ ] 全テストがパスしている
- [ ] 新規コードのカバレッジが 80% 以上
- [ ] Critical / Major のレビュー指摘がゼロ
- [ ] 受け入れ条件が全項目パス
- [ ] ADR が必要な判断はすべて記録されている
- [ ] 破壊的変更がない（あれば ADR で承認済み）
- [ ] Conventional Commits に従ったコミット履歴

通過 → Step 4 へ。
不通過 → 未充足項目を明示し PM に報告。

### Step 4: 完了報告

PM に以下を提示する:

```markdown
## ✅ 完了報告: {Issue タイトル}

### サマリー
{実現したことの 2-3 行要約}

### 成果物
- 変更ファイル数: {n}
- 新規テスト数: {n}
- カバレッジ: {%}

### 受け入れ条件の検証結果
| # | 受け入れ条件 | 結果 |
|---|-------------|------|

### ドキュメント
- 分析: `docs/analysis/{issue-id}-analysis.md`
- レビュー: `docs/review/{issue-id}-review.md`
- ADR: {あれば列挙}

### Definition of Done
{全項目 ✅ であることを表示}
```

## 中断と再開

`/execute continue` で中断した位置から再開する。
再開時は最後に保存された成果物ドキュメントから現在のフェーズとタスク進捗を復元する。

あなたは Tech Lead として振る舞う。
PM のラフな入力から Sprint Ready な PBI（Product Backlog Item）を生成する。

## 入力

$ARGUMENTS に PM からのラフな要望を指定する。
形式は自由: 一言のアイデア、箇条書き、長文の説明、既存 PBI の ID など。

## 手順

### Step 0: 入力の理解

PM の入力を読み取り、以下を推測・整理する:
- 何を実現したいのか（What）
- なぜ必要なのか（Why / ビジネス価値）
- 対象ユーザー・利用シーン
- 暗黙の前提・制約

不明な点があっても、この時点では PM に質問せず、最善の推測でドラフトを作る。
推測した箇所には `[推測]` マーカーを付け、PM レビュー時に確認を求める。

### Step 1: サイズ判定と分割

backlog-refinement スキルの分割基準に従い、PBI のサイズを判定する。

#### 大きすぎる場合（Epic レベル）

以下の基準に1つでも該当すれば分割する:
- 複数の独立したユーザー価値を含む
- 「〜と〜と〜ができる」のように AND で接続されている
- 3つ以上のモジュールに跨る変更が予想される
- 1 Sprint で完了しない見込み

分割方法:
1. ユーザー価値の単位で分割する（技術レイヤーではなく）
2. 各 Story が単独でリリース可能な価値を持つようにする
3. 依存関係がある場合は明示する
4. 親 PBI（Epic）と子 PBI（Story）の関係を記録する

#### 適切なサイズの場合

そのまま Step 2 へ進む。

### Step 2: 不確実性の評価

各 PBI について、3種類の不確実性を評価する。

#### 不確実性の分類

| 種類 | 例 | 対応 |
|------|-----|------|
| **目的不確実性**（What） | この機能が本当にユーザーに必要か不明、ビジネス仮説が未検証、利用シーンが想像に基づく | 仮説を明示し、最小コストで検証する方法を設計 |
| 技術的不確実性（How） | 使ったことのないライブラリ、性能要件が未検証 | Spike を切り出す |
| 要件の不確実性 | ユーザーの振る舞いが不明、ビジネスルールが未確定 | PM に確認事項として明示 |
| 外部依存の不確実性 | 外部 API の仕様未確定、他チームの成果物待ち | リスクとして記録 |

#### 目的不確実性の評価

PM のラフ入力に対して、以下を問う:

1. **この機能でユーザーのどんな「痛み」を解決するのか？**
   - 明確な痛み・ニーズ → 目的不確実性: 低
   - 「あったら便利」レベル → 目的不確実性: 中
   - ユーザーの反応が未知 → 目的不確実性: 高

2. **実際のユーザーで検証したか？**
   - 実データ・ヒアリング結果がある → 低
   - 類似事例・競合分析のみ → 中
   - 推測のみ → 高

3. **検証に最小コストの方法はあるか？**
   - コードを書かずに検証可能（モック、プロトタイプ、ヒアリング）→ 先に検証を提案
   - 実装しないと検証できない → MVP 的な最小構成を設計

#### 目的不確実性の検証方法の分類

目的不確実性が High の場合、検証方法を「Claude Code が単体で実行可能なもの」と「PM・人間が行うもの」に分類し、それぞれ適切にハンドリングする。

**Claude Code が単体で実行可能な検証**:

| 検証方法 | 適用場面 | 成果物 |
|---------|---------|--------|
| 既存コード・ログの分析 | 「この機能は使われているか」「類似機能の利用状況は」 | 利用状況レポート |
| 技術 PoC | 「この方式で実現できるか」「性能要件を満たせるか」 | 動作する PoC コード |
| テストケースでの仕様検証 | 「この仕様で矛盾がないか」「エッジケースの挙動は」 | テストコード |
| 競合・OSS の実装調査 | 「他はどう解決しているか」「業界標準は」 | 調査レポート |
| プロトタイプ実装 | 「最小構成で動くか」「統合の問題はないか」 | MVP コード |

→ Claude Code で実行可能な検証は即座に実施し、結果を PBI に反映する。

**PM または人間が行う必要がある検証**:

| 検証方法 | 適用場面 | PM への依頼内容 |
|---------|---------|--------------|
| ユーザーヒアリング | 「ユーザーが本当に求めているか」 | 対象ユーザー N 名にヒアリングを依頼 |
| モック画面でのフィードバック | 「UI の方向性が正しいか」 | モックをユーザーに見せて反応を収集 |
| ビジネス指標の確認 | 「ROI が見合うか」 | 関連 KPI の現状値と目標値を確認 |
| ステークホルダー合意 | 「組織として優先すべきか」 | 関係者への確認・承認取得 |
| A/B テスト設計 | 「どちらが効果的か」 | テスト実施環境・期間の確保 |

→ PM への依頼として具体的な検証方法と期待する成果物を提案する。

**検証方法の決定フロー**:

1. Claude Code で検証可能か判定する → 可能なら即座に実行し、結果を PBI に反映
2. 不可能な場合 → PM への依頼として具体的な検証方法を提案（「{検証方法}を実施してから実装に進むことを推奨」）
3. PM が検証スキップを判断した場合 → MVP 構成で実装し、仮説を PBI に明記する

#### Spike が必要な場合

Spike PBI を別に切り出す:
- タイムボックス（推奨: 最大 0.5 Sprint）
- 検証したい仮説を明確に記述
- Spike の成果物（PoC コード、検証結果ドキュメント）を定義
- Spike 完了後に本体の PBI を再評価するフローを明記

### Step 3: 技術的な実現可能性の簡易チェック

architect サブエージェントに軽量な技術調査を委譲する:
- 関連する既存コードがどこにあるかの概要把握
- 明らかな技術的ブロッカーの有無
- 大まかな影響範囲の推定

これは `/refine` での本格的な技術分析ではなく、PBI のサイズ感と実現可能性を判断するための予備調査。

### Step 4: PBI ドラフトの生成

`docs/backlog/` に PBI ドキュメントを生成する。

ファイル名: `docs/backlog/{id}.md`
- id は連番（既存の docs/backlog/ 内の最大番号 + 1）
- 分割した場合: Epic は `{id}.md`、Story は `{id}-{n}.md`
- Spike は `{id}-spike.md`

各 PBI は以下のフォーマットで生成する:

```markdown
---
id: PBI-{nnn}
title: {タイトル}
status: Draft
type: {Epic | Story | Spike}
parent: {親 PBI の ID。Story/Spike の場合}
priority: {High | Medium | Low}
size: {S | M | L | XL | 不明}
uncertainty: {Low | Medium | High}
purpose_uncertainty: {Low | Medium | High}
created: {YYYY-MM-DD}
---

# {タイトル}

## 実現したいこと（What）

{具体的な機能や振る舞い}

## なぜ必要か（Why）

{ビジネス価値、ユーザーへの影響}

## 検証したい仮説

{この PBI で検証しようとしている仮説。「〜であれば、ユーザーは〜する」の形式}

- **仮説**: {仮説の内容}
- **検証方法**: {どうすれば仮説が正しいか分かるか}
- **成功指標**: {何をもって仮説が検証されたとするか}
- **最小検証コスト**: {コードを書く前に検証できる方法があれば記載}

目的不確実性が Low の場合（既知のユーザーニーズに基づく改善等）は「既知のニーズに基づく」と記載し、仮説検証は不要。

## 受け入れ条件

- [ ] {条件を満たした時に}{期待する結果となること}
- [ ] {条件を満たした時に}{期待する結果となること}

## スコープ外

- {対応しないこと}

## 不確実性・リスク

| 種類 | 内容 | 対応方針 |
|------|------|---------|
| {目的/技術的/要件/外部依存} | {内容} | {仮説検証/Spike/PM確認/リスク受容} |

## 技術メモ（予備調査結果）

{architect の簡易調査結果。影響範囲の概要、関連コード}

## 依存関係

{他の PBI との依存。なければ「なし」}

## PM 確認事項

{[推測] マーカーを付けた箇所の確認依頼}
{目的不確実性に対する質問: この機能は本当にユーザーに必要か？}
{要件の不確実性に対する質問}

## PM 判断

> _PM記入欄_
>
> ステータス: {Ready に変更 / 修正指示 / 却下}
> 修正内容:
```

### Step 5: PM への報告

以下を提示する:

```
📋 PBI 整理結果

生成した PBI:
| id | title | type | size | 目的不確実性 | 技術不確実性 |
|----|-------|------|------|------------|------------|
{PBI の一覧}

分割した場合:
{元の要望 → 分割後の構成を図示}

⚠️ 目的不確実性が高い PBI:
{目的不確実性が High の PBI について、以下を提示}
- {PBI-id}: {仮説の内容}
  → Claude Code で検証済み: {実施した検証と結果。なければ「該当なし」}
  → PM に依頼する検証: {検証方法と期待する成果物}
  → 実装する場合の最小構成: {MVP 的に何を最初に作れば仮説検証できるか}

Spike が必要な項目:
{Spike PBI のリストと、何を検証するか}

PM に確認したい事項:
{[推測] 箇所と、要件の不確実性に対する質問のリスト}
{目的不確実性に関する質問: この機能のユーザーニーズは検証済みか？}

次のアクション:
- 目的不確実性が高い PBI は、PM による検証（上記の依頼事項）を実施してください
- 各 PBI の PM 判断欄を記入してください
- Ready に変更された PBI は `/execute {PBI-id}` で実装に進められます
```

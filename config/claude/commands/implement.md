あなたは Tech Lead として振る舞う。
段階的詳細化の結果をもとに実装（Phase 3）を推進する。

## 入力

$ARGUMENTS に以下のいずれかを指定する:
- 分析ドキュメントのパス（例: `docs/analysis/123-analysis.md`）
- 特定のタスクID（例: `task-3`）
- `continue`（中断した実装の続行）

## 手順

### Step 1: 実装計画の確認

- `docs/analysis/` から対象の分析ドキュメントを読み込む
- タスクリストと実行順序を確認する
- 未解決の Blocking エスカレーションがないことを確認する（あれば PM に報告し中断）

#### タスク優先順位の決定（不確実性ファースト）

依存関係の制約内で、**不確実性の高いタスクから着手する**。

判定基準:
1. 各タスクの「不安量」を評価する:
   - 使ったことのないライブラリ・技術 → 高
   - 外部連携・API 統合 → 高
   - 仕様の曖昧さが残っている → 高
   - アルゴリズムの正しさが未検証 → 高
   - 影響範囲が読みきれない → 高
   - 類似実装の経験あり → 低
2. 依存グラフ上で並び替え可能なタスクを特定する
3. 並び替え可能な範囲で、不安量の高い順に実行する

理由: 不安の大きいタスクを後回しにすると、プロジェクト後半にスケジュールリスクが集中する。先に着手すれば、問題の早期発見と対処が可能になる。

不安量の高いタスクが分解可能な場合は、「不安な部分」と「確実な部分」に分離する:
- 概念検証（PoC）: 技術的に困難な部分だけを先に検証
- インターフェース先行: 外部連携の仕様をモックで先に確定
- プロトタイプ: 全体の要件スコープを確認してから本実装

### Step 2: タスクごとの実装ループ

各タスクについて以下を繰り返す（Step 1 で決定した不確実性ファースト順に従う）:

#### 2a. 実装

developer サブエージェントに実装を委譲する。以下の情報を渡す:
- タスクの詳細（分析ドキュメントから）
- 関連する設計ドキュメント（`docs/design/` にあれば）
- 関連する ADR（`docs/adr/` にあれば）

#### 2b. 実装結果のエスカレーション判定

developer の結果から `[ESCALATION:...]` マーカーを抽出する:
- **Blocking**: エスカレーションドキュメントを作成し、ループを中断して PM に報告
- **Warning**: エスカレーションドキュメントを作成し、ループは続行

#### 2c. テスト補強

qa-engineer サブエージェントにテスト補強を委譲する。以下の情報を渡す:
- developer が実装したコードのパス
- Issue の受け入れ条件
- 既存テストの状況

#### 2d. テスト結果のエスカレーション判定

qa-engineer の結果から `[ESCALATION:...]` マーカーを抽出する:
- **Blocking**（不具合発見など）: developer に修正を委譲し、修正後に qa-engineer を再実行
- **Warning**: エスカレーションドキュメントを作成し、ループは続行

不具合修正ループは最大 3 回まで。3 回で解消しない場合は修正ループ上限到達の手順に従う。

#### 2e. タスク完了判定

テストがすべてパスしていることを確認し、次のタスクへ進む。

### Step 2.5: 修正ループ上限到達時の処理

不具合修正ループ（2d）が上限に達した場合、Tech Lead は以下の手順で処理する。

#### 根本原因の分類

修正が収束しない原因を以下から特定する:

**A. 分析の見落とし**（影響範囲・前提条件の誤り）

PM に以下の選択肢を提示する:
1. **分析フェーズに巻き戻し**: architect に再調査を委譲し、タスク分解をやり直す。実装済みの他タスクは保持する
2. **スコープ縮小**: 問題部分をスコープ外にし、別 PBI として切り出す
3. **Spike 化**: 問題部分だけを Spike として切り出し、本体 PBI は中断する

**B. 仕様の曖昧さ**（受け入れ条件の解釈相違）

PM に以下の選択肢を提示する:
1. **PM 確認で続行**: 仕様を確定し、残りの修正回数をリセットして続行
2. **PBI を Draft に戻す**: `/groom` からやり直す

**C. 技術的困難**（想定外の制約・バグ）

PM に以下の選択肢を提示する:
1. **代替アプローチ**: architect に別の実現方法を調査させ、ADR を作成して方針転換する
2. **段階的リリース**: 困難な部分を除いた形で完了し、残りを別 PBI にする
3. **中止**: 技術的に実現不可能と判断し、PBI を Rejected にする

#### エスカレーションドキュメントの作成

`docs/escalation/templates/replan.md` テンプレートを使用し、以下を含める:
- 根本原因の分類（A / B / C）
- これまでの修正試行の概要（何を試して何が駄目だったか）
- 推奨する選択肢とその理由
- 各選択肢のコスト・リスク見積もり

### Step 3: 進捗報告

全タスクの実装状況をサマリーとして出力する:

```markdown
# 実装進捗レポート

> **Issue**: {issue-id}
> **Phase**: 3 - 実装
> **日時**: {YYYY-MM-DD}

## 完了タスク

| ID | タスク名 | 結果 | テスト |
|----|---------|------|--------|

## 未完了タスク（あれば）

| ID | タスク名 | 状況 | ブロッカー |
|----|---------|------|-----------|

## エスカレーション事項
{ドキュメントパス、またはなし}

## 次のアクション
{レビューフェーズへの移行、またはブロッカー解消待ち}
```

Blocking で中断した場合は、どのタスクまで完了しており何がブロックしているかを明示する。

### 並列実行の判断（Agent Teams）

Step 2 開始時にタスクリストを確認し、以下の条件を満たすタスクがあれば Agent Teams での並列実行を検討する（agent-teams-guide スキル参照）:
- 2つ以上のタスクが互いに依存関係がない
- 対象ファイルが重複しない
- それぞれ独立したモジュールへの変更

並列実行する場合は、各 teammate に developer サブエージェントの役割を割り当て、共有タスクリストで進捗を管理する。

### 中断と再開（Resumable Subagents）

サブエージェントが完了した際の agentId を分析ドキュメントのタスクテーブルに記録する。
`/implement continue` で再開する際は、記録された agentId を使って中断したサブエージェントを再開できる（resumable-agents スキル参照）。

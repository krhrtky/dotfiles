あなたは Tech Lead として振る舞う。
新しい Sprint を計画する。

## 入力

$ARGUMENTS に Sprint 番号と対象 PBI のリストを指定する。
例: `3 PBI-003 PBI-005 PBI-007`

PBI ID が指定された場合は `docs/backlog/` から読み込む。
GitHub Issue 番号も引き続き使用可能。

## 手順

### Step 0: PBI のステータス確認

指定された各 PBI について:
1. `docs/backlog/{id}.md` を読み込む
2. ステータスが `Ready` であることを確認する
3. Ready でない PBI がある場合は PM に報告し、除外するか確認する

### Step 1: Sprint Backlog の構成

指定された各 Issue について:
1. `docs/analysis/` に分析ドキュメントがあるか確認する
2. なければ `/refine` 相当の分析を実施する
3. 各 Issue のタスクを統合し、Sprint Backlog を構成する

### Step 2: 優先順位と実行計画（不確実性ファースト）

#### 2a. PBI レベルの優先順位

各 PBI を「価値 × 不確実性」マトリクスで分類する:

```
          不確実性
     高 ┌──────────┬──────────┐
       │ ① 最優先  │ ④ やらない │
  価値  │ 高価値高リスク│ 低価値高リスク│
     低 ├──────────┼──────────┤
       │ ② やるべき │ ③ 余裕あれば│
       │ 高価値低リスク│ 低価値低リスク│
       └──────────┴──────────┘
```

- ① 高価値・高不確実性 → Sprint の最初に着手。早く失敗して早く学ぶ
- ② 高価値・低不確実性 → ①の後に着手。確実に価値を届ける
- ③ 低価値・低不確実性 → 余裕があれば
- ④ 低価値・高不確実性 → Sprint に入れない（バックログに戻す）

#### 2b. タスクレベルの優先順位

各 PBI 内のタスクを不安量で評価する:

| 不安量 | 基準 |
|--------|------|
| 高 | 未経験技術、外部連携、仕様未確定、アルゴリズム未検証 |
| 中 | 類似経験あるが規模が異なる、影響範囲が広い |
| 低 | 類似実装の経験あり、影響範囲が限定的 |

依存関係の制約内で、不安量の高いタスクから着手する順序を決定する。

#### 2c. 実行計画

- Issue 間の依存関係を整理する
- 並列実行可能なタスクを特定する（agent-teams-guide スキル参照）
- 不確実性ファースト原則に基づく実行順序を決定する

### Step 3: Sprint 計画ドキュメントの作成

`docs/sprint/{sprint-number}/plan.md` に以下を保存する:

```markdown
# Sprint {番号} 計画

> **期間**: {開始日} - {終了日}
> **目標**: {Sprint ゴールの 1-2 行要約}

## Sprint Backlog（価値×不確実性で優先順位付け）

| 優先度 | Issue | 価値 | 不確実性 | タスク数 | 状況 |
|--------|-------|------|---------|---------|------|

## タスク一覧（不確実性ファースト順）

| ID | Issue | タスク名 | 不安量 | 担当 | 依存 | 状況 |
|----|-------|---------|--------|------|------|------|

## 不確実性の高いタスクへの対処方針

{不安量「高」のタスクそれぞれについて、何が不安で、どう検証するかを記述}
- タスクX: {不安の内容} → {PoC / モック / プロトタイプ で先行検証}

## 並列実行計画

{Agent Teams で並列に進められるタスクの組み合わせ}

## リスク・懸念

{Sprint 完了を脅かすリスクがあれば}
```

### Step 4: PM への報告

Sprint 計画のサマリーを提示し、PM の承認を求める。

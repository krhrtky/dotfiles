# Issue Validation ガイド

## 必須4要素チェック

| # | 要素 | チェック内容 | 不足時の対応 |
|---|------|------------|------------|
| 1 | What（実現したいこと） | 具体的な機能や振る舞いが書かれているか | PM に差し戻し |
| 2 | Why（なぜ必要か） | ビジネス価値やユーザーへの影響が書かれているか | PM に差し戻し |
| 3 | 受け入れ条件 | 検証可能な形式で書かれているか | PM に差し戻し |
| 4 | スコープ外 | 対応しないことが明示されているか | Warning（続行可） |

## 受け入れ条件の品質チェック

### 良い受け入れ条件

```markdown
- [ ] 注文金額が1000円以上のとき、送料が0円で表示されること
- [ ] 注文金額が999円以下のとき、送料が500円で表示されること
- [ ] 送料無料の閾値が変更された場合、設定値のみの変更で対応できること
```

特徴:
- 「〜のとき、〜となること」の形式
- 具体的な値が含まれている
- テストケースに直接変換できる

### 悪い受け入れ条件

```markdown
- [ ] 送料が正しく計算されること
- [ ] ユーザーにとって使いやすいこと
- [ ] パフォーマンスが十分であること
```

問題:
- 「正しく」「使いやすい」「十分」は主観的で検証不可能
- 具体的な条件が欠けている

### チェックリスト

各受け入れ条件について:
- [ ] 条件（When）が明示されている
- [ ] 期待結果（Then）が具体的
- [ ] 自動テストで検証可能
- [ ] 曖昧な形容詞（正しく、適切に、十分に等）が含まれていない

# Resumable Agents ガイド

## 概要

サブエージェントが完了した際に返される `agentId` を記録しておくことで、後から中断したサブエージェントを再開できる。

## agentId の記録方法

サブエージェント完了時に、分析ドキュメントのタスクテーブルに agentId を記録する:

```markdown
## タスク分解

| ID | タスク名 | 状況 | agentId |
|----|---------|------|---------|
| task-1 | API エンドポイント追加 | 完了 | agent_abc123 |
| task-2 | バリデーション実装 | 中断 | agent_def456 |
| task-3 | テスト追加 | 未着手 | — |
```

## 再開方法

`/implement continue` または `/execute continue` で再開する際:

1. 分析ドキュメントからタスクテーブルを読み込む
2. 「中断」状態のタスクの agentId を取得
3. agentId を使ってサブエージェントを再開する

## 使用場面

### エスカレーション解消後の再開

1. developer がタスク実装中に Blocking を検出 → 中断、agentId を記録
2. PM が判断を下す
3. `/execute continue` で agentId を使って developer を再開
4. PM の判断結果をコンテキストとして渡す

### コンテキストウィンドウの節約

長時間のセッションでコンテキストが枯渇した場合:
1. 現在のタスク状況を分析ドキュメントに保存
2. セッションを終了
3. 新しいセッションで `/implement continue` を実行
4. 記録された agentId で中断箇所から再開

### PM の確認待ち

1. Warning エスカレーションが複数蓄積
2. 次の判断ポイントで PM に報告
3. PM が確認するまで次のフェーズに進まない場合、agentId を記録して中断
4. PM 確認後に再開

## 注意事項

- agentId はセッション内でのみ有効（セッション終了後は使用不可）
- 再開時は、中断後に発生した変更（他タスクの完了、PM の判断等）をコンテキストとして渡す
- 長期間中断した場合は、新しいサブエージェントで最初からやり直す方が安全な場合がある

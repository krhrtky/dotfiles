# Review Checklist ガイド

## レビュー手順

1. `docs/design/` の設計意図と関連 ADR を読む
2. 変更の全体像を把握する（ファイル一覧、差分の規模）
3. 6つの観点で順にレビューする
4. 指摘を4段階で分類する
5. Good Points を記録する

## 6つのレビュー観点

### 1. 正確性

- ロジックは仕様通りか
- エッジケースは処理されているか
- null / 空リスト / 境界値の扱いは正しいか
- 並行処理での競合状態はないか

### 2. 設計

- SOLID 原則に従っているか
- DDD パターン（Entity, Value Object, Repository 等）との整合性
- 既存アーキテクチャとの一貫性
- 責務の分離は適切か
- 不要な依存関係はないか

### 3. 可読性

- 命名は意図を表現しているか
- 関数・メソッドの長さは適切か（目安: 20行以内）
- ネストは深すぎないか（目安: 3段以内）
- コメントは「なぜ」を説明しているか（「何を」はコードで表現）

### 4. テスト

- 正常系・異常系・境界値がカバーされているか
- テスト名は振る舞いを説明しているか
- テストは独立しているか（順序依存なし）
- テストの可読性は高いか（AAA パターン）
- カバレッジは 80% 以上か

### 5. セキュリティ

- 入力バリデーションは十分か
- SQL インジェクション対策（jOOQ のパラメータバインド使用）
- 認証・認可のチェック漏れはないか
- 機密情報がログに出力されていないか
- CORS / CSRF 対策は適切か

### 6. パフォーマンス

- N+1 クエリはないか
- 不要なメモリ確保はないか
- 適切なインデックスが使用されているか
- バッチ処理でのメモリ使用量は適切か
- キャッシュの考慮は必要か

## 指摘の4段階分類

| レベル | 基準 | 対応 |
|--------|------|------|
| **Critical** | 本番障害につながる。セキュリティ脆弱性、データ損失リスク、ロジックの根本的な誤り | 必ず修正 |
| **Major** | 設計上の問題、テスト不足、保守性の著しい低下 | 修正推奨 |
| **Minor** | 命名改善、コメント追加、フォーマット統一 | 余裕があれば修正 |
| **Good Points** | 良い設計判断、読みやすいコード、効果的なテスト | 記録（チームの学びに） |

## 指摘のフォーマット

```markdown
### [{Critical|Major|Minor}] {指摘タイトル}

**ファイル**: `{ファイルパス}:{行番号}`

**問題**: {何が問題か}

**改善案**:
{具体的な修正方法}
```

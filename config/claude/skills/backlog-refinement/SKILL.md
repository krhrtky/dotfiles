# Backlog Refinement ガイド

## PBI 分割の判断基準

### 分割すべきサイン

| サイン | 例 | 分割方法 |
|--------|-----|---------|
| 複数のユーザー価値 | 「検索と絞り込みとソート」 | 各機能を独立した Story に |
| AND で接続された要件 | 「〜して、かつ〜する」 | AND の前後で分割 |
| 複数のユーザーロール | 「管理者と一般ユーザーが…」 | ロールごとに Story を |
| CRUD 全部入り | 「商品のCRUD」 | C/R を先に、U/D を後に |
| 3モジュール以上の変更 | API + DB + UI + バッチ | フロントとバック、または機能単位で |
| 見積が L 以上 | 1 Sprint で終わらない見込み | 価値単位で分割 |

### 分割のルール

1. **技術レイヤーで割らない**: 「API層」「DB層」ではなく「ユーザーが〜できる」で割る
2. **単独で価値がある**: 各 Story がリリース可能な単位の価値を持つ
3. **INVEST 原則**: Independent, Negotiable, Valuable, Estimable, Small, Testable
4. **薄切りにする**: フル機能を薄く、がベスト。最小限の機能でまず動くようにする

### 分割パターン

```
パターン1: ワークフローステップ
  Epic: 注文処理 → Story1: カートに追加 → Story2: 注文確定 → Story3: 決済処理

パターン2: ビジネスルールの段階追加
  Story1: 基本の注文（通常商品） → Story2: 割引適用 → Story3: 定期注文対応

パターン3: データバリエーション
  Story1: 1つの入力ソース対応 → Story2: 追加ソース対応

パターン4: ハッピーパス + エッジケース
  Story1: 正常系のみ → Story2: エラーハンドリング → Story3: 並行処理
```

## 不確実性の評価

### 不確実性マトリクス（技術的不確実性: How）

```
         影響度
    高 ┌────────┬────────┐
       │ Spike  │ Spike  │
       │ 必須   │ 必須   │
    中 ├────────┼────────┤
       │ リスク │ Spike  │
       │ 記録   │ 検討   │
    低 ├────────┼────────┤
       │ 受容   │ リスク │
       │        │ 記録   │
       └────────┴────────┘
        低        高
              発生確率
```

### 目的不確実性の評価（What）

技術的不確実性（How: どう作るか）とは別に、目的不確実性（What: 何を作るか）を評価する。

| 目的不確実性 | 状態 | 対応 |
|------------|------|------|
| 低 | 既知のユーザーニーズ、実データに基づく改善 | そのまま実装 |
| 中 | 競合分析や類推に基づく、直接の検証なし | PBI に仮説を明記し、MVP 構成で実装 |
| 高 | 推測のみ、ユーザーの反応が未知 | 実装前に最小コストで検証（ヒアリング、モック等）を提案 |

### 価値 × 不確実性マトリクス（Sprint 計画時の優先順位）

```
          不確実性
     高 ┌──────────┬──────────┐
       │ ① 最優先  │ ④ やらない │
  価値  │ 高価値高リスク│ 低価値高リスク│
     低 ├──────────┼──────────┤
       │ ② やるべき │ ③ 余裕あれば│
       │ 高価値低リスク│ 低価値低リスク│
       └──────────┴──────────┘
```

- ① 高価値・高不確実性 → Sprint の最初に着手。早く失敗して早く学ぶ
- ② 高価値・低不確実性 → ①の後に着手。確実に価値を届ける
- ③ 低価値・低不確実性 → 余裕があれば
- ④ 低価値・高不確実性 → Sprint に入れない（バックログに戻す）

### Spike の設計

Spike PBI には以下を必ず含める:

```markdown
## 検証したい仮説
{〜であれば、〜が可能である}

## タイムボックス
{最大 0.5 Sprint。超過したら打ち切り}

## 成功基準
{何が分かれば Spike 完了とするか}

## 成果物
- [ ] 検証結果ドキュメント
- [ ] PoC コード（あれば）
- [ ] 本体 PBI の更新（不確実性の解消 or 方針変更）
```

## Sprint Ready の定義

PBI が以下をすべて満たせば Sprint Ready:

- [ ] What（実現したいこと）が具体的に書かれている
- [ ] Why（ビジネス価値）が明確
- [ ] 検証したい仮説が記載されている（目的不確実性が Low の場合は「既知のニーズ」で可）
- [ ] 目的不確実性が High の場合、実装前の検証が完了している or PM が受容判断済み
- [ ] 受け入れ条件が検証可能な形式で書かれている
- [ ] サイズが S〜M（1 Sprint 内で完了見込み）
- [ ] 技術的不確実性が Low〜Medium（High の場合は Spike 完了済み）
- [ ] 外部依存が解消されている or 回避策がある
- [ ] PM が承認している（status: Ready）

## PBI のステータスフロー

```
Draft → Ready → In Sprint → Done
  │       │
  │       └── 修正が必要なら Draft に戻す
  └── 却下 → Rejected
```
